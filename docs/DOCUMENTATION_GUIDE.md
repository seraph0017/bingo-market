## 一、文档目的

本文档定义 Bingo Market 项目的技术文档编写规范，确保：

- **一致性**：所有文档遵循统一的结构和格式
- **可读性**：文档清晰易懂，便于团队成员阅读
- **可维护性**：便于更新和长期维护
- **可追溯性**：版本变更有记录，历史可追溯

## 三、文档结构规范

### 3.1 标准文档结构

所有技术文档应遵循以下基本结构：

```markdown
# 文档标题

| 元数据 | 值 |
|--------|-----|
| **版本** | v1.0 |
| **日期** | YYYY-MM-DD |
| **作者** | 作者名 |
| **状态** | 草稿/评审中/已批准/已归档 |
| **维护者** | Tech Doc Specialist |

---

## 修订历史

| 版本 | 日期 | 作者 | 变更说明 |
|------|------|------|----------|
| v1.0 | YYYY-MM-DD | 作者 | 初始版本 |

---

## 目录

1. [章节 1](#章节 -1)
2. [章节 2](#章节 -2)
...

---

## 1. 概述/引言

文档目的、范围、背景信息。

## 2. 正文内容

根据文档类型组织内容。

## 3. 附录/参考

相关链接、术语表等。
```

### 3.2 元数据要求

**必填字段**：
- `版本`：遵循语义化版本号 (见第 6 节)
- `日期`：YYYY-MM-DD 格式
- `作者`：文档创建者
- `维护者`：固定为 `Tech Doc Specialist`

**选填字段**：
- `状态`：草稿/评审中/已批准/已归档
- `审核人`：文档审核者姓名
- `批准人`：文档批准者姓名

## 四、格式规范

### 4.1 标题层级

```markdown
# 一级标题（文档标题，仅使用一次）
## 二级标题（主要章节）
### 三级标题（子章节）
#### 四级标题（详细内容）
```

**规则**：
- 一级标题仅用于文档标题
- 二级标题用于主要章节，建议使用中文数字编号（一、二、三...）
- 三级标题用于子章节，建议使用阿拉伯数字（1.1, 1.2...）
- 避免超过四级标题

### 4.2 目录（TOC）

**要求**：
- 超过 3 个主要章节的文档必须包含目录
- 目录应放在修订历史之后、正文之前
- 使用 Markdown 锚点链接

**示例**：
```markdown
## 目录

1. [概述](#一概述)
2. [功能需求](#二功能需求)
3. [技术实现](#三技术实现)
4. [附录](#四附录)

---
```

### 4.3 代码块

**要求**：
- 所有代码块必须指定语言标识符
- 长代码块应有注释或说明

**示例**：
````markdown
### Python 示例
```python
from fastapi import FastAPI

app = FastAPI(title="Bingo Market API")
```

### JSON 响应
```json
{
  "status": "success",
  "data": {
    "balance": 100000
  }
}
```

### Bash 命令
```bash
cd code/backend
python -m uvicorn app.main:app --reload
```
````

### 4.4 表格

**要求**：
- 表格必须有表头
- 复杂表格应有说明文字
- 表格内容应对齐（左对齐或居中对齐）

**示例**：
```markdown
### API 参数说明

| 参数 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| user_id | string | 是 | - | 用户唯一标识 |
| amount | integer | 是 | - | 充值金额（VND） |
| note | string | 否 | "" | 备注信息 |
```

### 4.5 列表

**要求**：
- 有序列表用于步骤、优先级等有序内容
- 无序列表用于选项、特征等无序内容
- 嵌套列表不超过三层

**示例**：
```markdown
### 有序列表（步骤）
1. 第一步：准备工作
2. 第二步：执行操作
3. 第三步：验证结果

### 无序列表（特征）
- 特征一
- 特征二
  - 子特征
  - 子特征
```

### 4.6 强调和重点

| 用途 | 语法 | 示例 |
|------|------|------|
| 重要提示 | `**文本**` | **重要：必须先完成 KYC** |
| 警告/危险 | `> **警告**：文本` | > **警告**：此操作不可逆 |
| 引用 | `> 文本` | > 根据 PRD 第 3.2 节 |
| 行内代码 | `` `文本` `` | 调用 `/api/v1/wallet` 接口 |

### 4.7 链接

**内部链接**：
```markdown
详见 [API 文档](./api/wallet-api.md) 第 3 节。
参考 [项目结构](./PROJECT_STRUCTURE.md)。
```

**外部链接**：
```markdown

参考 [Vue 3 文档](https://vuejs.org/)。
```

**图片**：

## 六、版本控制

### 6.1 版本号规则

采用语义化版本号：`主版本号。次版本号。修订号`

| 变更类型 | 版本号变更 | 示例 |
|----------|-----------|------|
| 重大架构变更 | 主版本号 +1 | v1.0 → v2.0 |
| 新功能添加 | 次版本号 +1 | v1.0 → v1.1 |
| 错误修复/小修改 | 修订号 +1 | v1.0.0 → v1.0.1 |

### 6.2 变更说明规范

修订历史中的变更说明应清晰描述变更内容：

**好的示例**：
- `添加钱包模块 API 文档`
- `修复 JWT 认证流程描述错误`
- `更新越南限额说明`

**不好的示例**：
- `更新文档`（过于模糊）
- `修改`（无意义）

### 6.3 文档状态流转

```
草稿 → 评审中 → 已批准 → (发布) → 已归档
         ↓
       驳回 → 草稿
```

| 状态 | 说明 | 操作权限 |
|------|------|----------|
| 草稿 | 文档编写中 | 作者可编辑 |
| 评审中 | 等待审核 | 仅审核人可评论 |
| 已批准 | 审核通过 | 冻结，仅可小修 |
| 已归档 | 历史版本 | 只读 |

## 七、审查流程

### 7.1 审查清单

文档发布前必须完成以下检查：

- [ ] 文档结构符合本规范第 3 节要求
- [ ] 包含元数据表格和修订历史
- [ ] 包含目录（超过 3 个章节时）
- [ ] 标题层级正确，无跳跃
- [ ] 代码块有语言标识符
- [ ] 表格有表头和说明
- [ ] 术语使用符合第 5 节规范
- [ ] 内部链接有效
- [ ] 版本号正确更新
- [ ] 变更说明清晰

### 7.2 审查流程

```
作者创建 → 自检 → 提交审查 → 同行评审 → 技术负责人批准 → 发布
```

### 7.3 审查角色

| 角色 | 职责 | 人员 |
|------|------|------|
| 作者 | 文档编写、自检 | 开发者/产品经理 |
| 审查人 | 内容准确性审查 | 技术负责人 |
| 批准人 | 最终批准发布 | 项目经理 |

## 八、维护流程

### 8.1 文档更新触发条件

以下情况需要更新文档：

1. **功能变更**：API 接口修改、业务逻辑变更
2. **Bug 修复**：文档中的错误信息
3. **架构调整**：系统架构、目录结构变更
4. **定期审查**：每季度进行一次文档健康检查

### 8.2 更新流程

1. 确定需要更新的文档
2. 评估变更影响范围
3. 更新文档内容和版本号
4. 在修订历史中记录变更
5. 通知相关干系人

### 8.3 文档归档

以下情况文档应归档：

- 功能已下线
- 方案已被替代
- 项目阶段结束

归档操作：
1. 在文档顶部添加归档标识
2. 移动到 `docs/archive/` 目录
3. 在索引中标记为已归档
